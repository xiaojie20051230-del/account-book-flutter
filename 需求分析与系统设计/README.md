# 文档标准

本文档目录包含随手记 App 的所有设计和架构文档。每份文档必须遵循以下约定。

---

## 1. 文档分类

| 类型 | 前缀 | 用途 | 必需章节 |
|------|--------|-------|---------|
| **ADR** | `ADR-` | 架构决策记录 — 记录设计决策及其上下文、可选方案和理由 | 上下文、决策、影响、可选方案 |
| **BUGFIX** | `BUGFIX-` | 批量 bug 修复记录 — 问题、根因、方案和验收标准 | 需求、UI/数据流、组件设计、验收标准、开放问题 |
| **设计规格** | `DESIGN-` | 新功能或组件的规格说明书 | 需求、UI/数据流、组件设计、验收标准、开放问题 |
| **指南** | `GUIDE-` | 环境搭建、代码规范、团队协作等辅助文档 | 视内容而定 |
| **测试方案** | `TEST-PLAN-` | 测试计划和测试用例 | 测试策略、白盒测试、黑盒测试、边界条件、交付物 |

---

## 2. 文件命名规则

```
{类型}-{NNN}-{分类描述}.md
  │       │      │
  │       │      └── 小写，连字符分隔，英文
  │       │          （如 trash-and-tracking）
  │       │
  │       └── 3 位零填充序号（001, 002, ..., 999）
  │
  └── 文档类型前缀（ADR / BUGFIX / DESIGN / GUIDE / TEST-PLAN）
```

**示例：**
- `ADR-001-interface-driven-architecture.md`
- `BUGFIX-001-bug-fixes.md`
- `DESIGN-001-mvp-architecture.md`
- `TEST-PLAN-001-mvp-test-plan.md`

**规则：**
1. 每种类型的序号唯一
2. 描述字段小写、连字符分隔，最长 50 字符
3. 文件名不含中文字符；文档正文允许中文
4. 废弃文档移至 `docs/archive/`，不删除

---

## 3. 元数据头部

每份文档必须以 **YAML 格式**的元数据开头：

```markdown
---
title: 文档标题
status: Proposed / Accepted / Deprecated / Superseded
author: 作者名
date: YYYY-MM-DD
version: 1.0
revision_history:
  - 1.0 — 初始版本
---
```

**状态生命周期：** `Proposed → Accepted → (Deprecated | Superseded)`

---

## 4. 各文档类型的必需章节

### 设计规格（DESIGN）

```markdown
## 1. 需求
## 2. UI / 数据流
## 3. 组件设计
## 4. 验收标准
## 5. 开放问题
```

### Bug 修复（BUGFIX）

与 DESIGN 相同的章节结构，但内容聚焦于问题根因分析和修复方案：

```markdown
## 1. 需求 — 问题清单（编号、现象、根因）
## 2. UI / 数据流 — 受影响流程的变更说明
## 3. 组件设计 — 文件修改清单和代码变更详情
## 4. 验收标准
## 5. 开放问题
```

### 测试方案（TEST-PLAN）

```markdown
## 1. 测试策略概览
## 2. 白盒测试（自动化）
## 3. 黑盒测试（手动）
## 4. 边界条件
## 5. 测试命令
## 6. 交付物清单
```

---

## 5. 写作风格

1. **语言：** 元数据和结构内容用英文；中文可用于说明性章节
2. **语气：** 指令性和事实性。避免"我们可能"、"也许"
3. **代码引用：** 始终包含文件路径和行号
4. **交叉引用：** 通过文档 ID 链接相关文档：`参见 DESIGN-003`

---

## 6. 版本控制

1. 每份文档在元数据中有一个"修订历史"字段
2. 重大变更（新增章节、改变决策）增加次版本号：1.0 → 1.1
3. 结构性重写增加主版本号：1.x → 2.0
4. 处于 `Proposed` 状态的文档在 `Accepted` 之前可无需版本号更新
